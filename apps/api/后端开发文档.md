# 后端开发文档

## 1. 文档目标

本文档用于说明 apps/api 当前后端服务的职责、接口定义、请求处理链路，以及如何配合前端公开接口完成联调和业务扩展。

当前后端是一个轻量级 Node.js HTTP 服务，重点不是框架复杂度，而是给宿主前端提供稳定、简单、可替换的接口边界。

## 2. 服务概览

后端入口文件：server.mjs

当前提供 3 类能力：

1. 服务健康检查。
2. 项目场景读取与保存。
3. 单个构件能耗数据查询。

数据存储方式：

1. 场景数据以 JSON 文件方式存放在 data/projects 目录。
2. 能耗数据当前为基于 projectId 和 componentId 计算出的 mock 数据。

## 3. 接口清单

### 3.1 GET /health

作用：

1. 提供服务存活检查。
2. 便于前端启动时探测后端可用性。

返回示例：

```json
{
  "status": "ok",
  "app": "editor-api",
  "timestamp": "2026-04-13T12:00:00.000Z"
}
```

### 3.2 GET /projects/:projectId/scene

作用：

1. 根据项目 ID 读取场景文件。
2. 供前端 ModelingEditorCoreModule 的 onLoad 回调使用。

参数说明：

1. projectId 仅允许 a-z、A-Z、0-9、_、-。

成功返回：

```json
{
  "projectId": "local-editor",
  "scene": {
    "nodes": {},
    "rootNodeIds": []
  },
  "updatedAt": "2026-04-13T12:00:00.000Z"
}
```

异常处理：

1. 项目文件不存在时返回 404。
2. projectId 非法时返回 400。

### 3.3 PUT /projects/:projectId/scene

作用：

1. 保存当前前端编辑后的 scene graph。
2. 供前端 ModelingEditorCoreModule 的 onSave 回调使用。

请求体结构：

```json
{
  "scene": {
    "nodes": {},
    "rootNodeIds": []
  }
}
```

校验规则：

1. body.scene 必须存在。
2. scene.nodes 必须是对象。
3. scene.rootNodeIds 必须是数组。

保存结果：

后端会写入 data/projects/:projectId.json，并附带 updatedAt 时间戳。

### 3.4 GET /projects/:projectId/energy/components/:componentId

作用：

1. 根据当前选中的构件查询能耗信息。
2. 为前端右侧业务面板提供演示数据。

参数校验：

1. projectId 仅允许 a-z、A-Z、0-9、_、-。
2. componentId 仅允许 a-z、A-Z、0-9、_、:、-。

返回结构：

```json
{
  "projectId": "local-editor",
  "componentId": "item_xxx",
  "binding": {
    "bindingType": "mock-meter",
    "bindingTargetId": "meter-item_xxx"
  },
  "currentPower": 3.4,
  "todayUsage": 21.5,
  "monthUsage": 301.8,
  "series": [
    { "time": "00:00", "value": 1.2 }
  ],
  "updatedAt": "2026-04-13T12:00:00.000Z"
}
```

说明：

当前并未接真实能耗平台，而是通过 seededNumber 和 buildEnergyPayload 生成稳定 mock 数据，方便前端联调与演示。

## 4. 请求处理流程

当前请求处理流程如下：

1. createServer 接收请求。
2. 统一处理 OPTIONS 预检请求，并返回 CORS 头。
3. 根据 pathname 匹配 health、scene、energy 三类路由。
4. 执行参数合法性校验。
5. 根据路由调用读文件、写文件或 mock 生成逻辑。
6. 使用 sendJson 统一输出响应头和 JSON 内容。

## 5. 后端如何处理前端请求

### 5.1 场景加载

前端通过 createEditorApiClient 调用 GET /projects/:projectId/scene。

后端处理步骤：

1. 校验 projectId。
2. 拼接 data/projects/:projectId.json 路径。
3. 读取文件并解析 JSON。
4. 将完整 payload 返回前端。

### 5.2 场景保存

前端通过 createEditorApiClient 调用 PUT /projects/:projectId/scene。

后端处理步骤：

1. 读取请求体。
2. 校验 scene 结构是否合法。
3. 自动创建目录。
4. 将 scene 与 updatedAt 一并写入 JSON 文件。
5. 返回保存后的 payload。

### 5.3 构件能耗查询

前端通过 loadComponentEnergy 调用 GET /projects/:projectId/energy/components/:componentId。

后端处理步骤：

1. 校验 projectId 和 componentId。
2. 使用 seededNumber 对输入做稳定哈希。
3. 生成 currentPower、todayUsage、monthUsage 和 series。
4. 返回可直接供前端右侧面板消费的 JSON。

## 6. 如何配合前端公开接口工作

当前前端公开接口主要有两类：

1. 建模场景接口：由 createEditorApiClient 消费。
2. 能耗查询接口：由 loadComponentEnergy 消费。

后端配合方式：

1. 如果需要替换场景存储介质，只要保持 GET/PUT /projects/:projectId/scene 的响应结构兼容，前端无需改动页面层。
2. 如果需要替换能耗数据源，只要保持 GET /projects/:projectId/energy/components/:componentId 的响应字段兼容，前端右侧面板无需重写。
3. 如果要新增业务，建议新增新路由并由对应前端 feature 自己封装调用，不要混入旧接口中。

## 7. 后续演进建议

当前 server.mjs 仍然是单文件结构。若后端继续增长，建议演进为：

```text
apps/api/
├── routes/
├── services/
├── validators/
├── data/
└── server.mjs
```

推荐拆分方向：

1. routes：按 health、scene、energy 拆路由。
2. services：封装文件存储和业务计算。
3. validators：集中校验 projectId、componentId 和 scene payload。
4. types：为前后端共享响应结构做准备。

## 8. 联调建议

1. 先启动 apps/api，再启动 apps/editor。
2. 在前端环境变量中设置 NEXT_PUBLIC_EDITOR_API_BASE_URL。
3. 先用 /health 检查后端可用，再验证场景加载与保存，最后验证能耗接口。
