# 前端接口说明

本文档面向宿主前端开发者，说明 apps/editor 当前最核心的公开接口、类型和调用方式。这里的“接口”既包括代码层公开的 TypeScript 接口，也包括前端对后端的访问封装。

## 一、页面装配接口

### 1. HostWorkbenchProps

位置：features/host-shell/components/host-workbench.tsx

作用：定义宿主工作台的页面级入参，目前核心参数是后端地址。

```ts
export interface HostWorkbenchProps {
  apiBaseUrl?: string
}
```

使用方式：
```tsx
<HostWorkbench apiBaseUrl={process.env.NEXT_PUBLIC_EDITOR_API_BASE_URL} />
```

适用场景：

1. 由 Next.js 页面入口注入后端服务地址。
2. 未来切换测试环境、预发环境或正式环境时，无需改业务代码，只改环境变量。

### 2. ModelingEditorCoreModule 宿主桥接回调

来源：@pascal-app/editor/modeling

当前宿主前端主要使用以下回调：

1. onLoad：加载场景。
2. onSave：保存场景。
3. onSelectionChange：获取当前选中构件。
4. onSaveStatusChange：同步保存状态到业务面板。

示例：

```tsx
<ModelingEditorCoreModule
  onLoad={handleLoad}
  onSave={handleSave}
  onSelectionChange={setSelection}
  onSaveStatusChange={setSaveStatus}
/>
```

作用：

1. 把编辑器核心与宿主前端解耦。
2. 让业务模块通过 selection snapshot 感知当前上下文，而不是侵入编辑器内部状态。

## 二、业务查询接口

### 1. HostQueryFilters

位置：features/energy-insights/lib/host-query.ts

作用：定义右侧查询筛选条件。

```ts
export interface HostQueryFilters {
  keyword: string
  levelId: string
  zoneId: string
  timeRange: string
  energyLevel: string
}
```

使用方式：

```ts
const [filters, setFilters] = useState<HostQueryFilters>({
  keyword: '',
  levelId: '',
  zoneId: '',
  timeRange: '24h',
  energyLevel: '',
})
```

作用说明：

1. 前端统一维护业务筛选状态。
2. 当前由前端演示聚合模型直接消费。
3. 后续如果切到真实批量查询接口，可以保持筛选结构不变，仅替换数据来源。

### 2. buildHostQueryModel

位置：features/energy-insights/lib/host-query.ts

签名：

```ts
buildHostQueryModel(nodes, filters): HostQueryModel
```

作用：

1. 从场景节点中提取楼层、房间和可筛选结果。
2. 为当前宿主右侧面板生成 levelOptions、zoneOptions 和 results。
3. 当前是演示级聚合实现，后续可被真实后端查询替换。

返回值：

```ts
export interface HostQueryModel {
  levelOptions: HostFilterOption[]
  zoneOptions: HostFilterOption[]
  results: HostQueryResult[]
}
```

### 3. loadComponentEnergy

位置：features/energy-insights/lib/energy-api.ts

签名：

```ts
loadComponentEnergy(baseUrl, projectId, componentId): Promise<EnergyApiResponse | null>
```

作用：

1. 根据当前选中的 componentId 请求后端能耗数据。
2. 当未配置 baseUrl 或接口返回 404 时返回 null，避免界面直接抛错。
3. 把前端业务 API 封装在 feature 内，避免泄漏到底层页面壳。

返回结构：

```ts
export interface EnergyApiResponse {
  projectId: string
  componentId: string
  currentPower: number
  todayUsage: number
  monthUsage: number
  series: { time: string; value: number }[]
  binding?: {
    bindingType: string
    bindingTargetId: string
  }
  updatedAt?: string
}
```

### 4. HostRightRailProps

位置：features/energy-insights/components/host-right-rail.tsx

作用：定义右侧业务面板对宿主壳的依赖关系，包括：

1. 当前选择结果。
2. 能耗数据加载状态。
3. 查询结果集。
4. 右侧面板宽度与折叠控制。

这组 props 的意义在于：把“数据”和“UI”分离，未来你可以用新的业务数据源继续复用右侧容器结构。

### 5. HostFilterBarProps

位置：features/energy-insights/components/host-filter-bar.tsx

作用：定义筛选栏组件所需入参。

核心点：

1. 只消费筛选值和候选项，不依赖编辑器内部 store。
2. 通过 variant 区分悬浮模式和侧栏模式。
3. 适合在未来迁移到其他业务面板中复用。

## 三、与后端交互的公开约定

当前前端默认依赖以下 HTTP 接口：

1. GET /health：服务健康检查。
2. GET /projects/:projectId/scene：加载场景。
3. PUT /projects/:projectId/scene：保存场景。
4. GET /projects/:projectId/energy/components/:componentId：加载单个构件的能耗信息。

其中：

1. 场景加载与保存通过 createEditorApiClient 进入建模核心模块。
2. 能耗查询通过 loadComponentEnergy 进入右侧业务模块。

## 四、推荐接入方式

如果要新增前端业务，请按以下步骤：

1. 在 features 下创建新业务目录。
2. 在该目录下定义自己的 types、services、components。
3. 如果新业务需要面板，优先在 host-shell 层挂载，不要修改 packages/editor。
4. 如果要复用当前右侧面板结构，可以新增自己的 xxx-right-rail 组件，而不是继续往 host-right-rail 中累加杂项逻辑。

## 五、常见扩展点

1. 替换演示查询模型：将 buildHostQueryModel 改为真实接口查询结果映射。
2. 替换能耗接口：保持 EnergyApiResponse 兼容或在 feature 内做适配层。
3. 扩展宿主面板：在 host-shell 中新增新的业务 feature 并排或切换展示。
